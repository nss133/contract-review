# Phase 1.5 재설계: 증적 중심 원자 체크리스트

- 작성일: 2026-07-07 · 상태: 승인됨
- 배경: Phase 1 사용자 피드백 — "LLM 코멘트가 중심이 되면 안 됨. 구체적 근거(가이드·법령·행정규칙 원문)가 각 체크 항목에 붙어야 하고, 원문을 바로 띄워볼 수 있어야 함. 항목은 단위 단위로 쪼갤 것."

## 설계 원칙 (Phase 1 대비 변경)

| | Phase 1 (폐기) | Phase 1.5 (채택) |
|---|---|---|
| 항목 단위 | 쟁점 묶음 (OUT-09에 7개 요건) | **조문 항·호 = 1 체크 항목** |
| 본문 | guidance 서술형 코멘트 | **check 질문 1문장 + 원문 quote** |
| 근거 | 조문 전체를 펼침으로 첨부 | **해당 항·호 문언 quote를 항목에 직접 표시** + 전체 원문 펼침 |
| 근거 없는 서술 | 허용됨 | **금지 — quote가 DB 원문과 대조 검증되지 않으면 빌드 경고 + 배지** |
| UI 축 | 조항별 카드 뷰 중심 | **체크리스트 표 중심**, 조항 매핑은 보조 배지 |

## 스키마 v2 (checkpoints → checks)

```yaml
checks:
  - id: OUT-09-1              # 전역 유일, 원자 단위
    check: 위탁 문서에 "위탁업무 수행 목적 외 개인정보의 처리 금지" 사항이 포함되어 있는가
    module: M-PRIV             # 선택 (유형 기본이면 생략)
    severity: 필수              # 필수/권장/참고
    norm_type: 강행             # 강행/임의/추정/간주/실무 (guidance 병기 → 명시 필드로 승격)
    basis: statute             # statute(법령 요건) | practice(실무 항목)
    sources:                   # basis=statute면 1개 이상 필수, 첫 항목에 quote 필수
      - law: 개인정보 보호법
        article: 제26조
        clause: 제1항 제1호     # 표시용
        quote: 위탁업무 수행 목적 외 개인정보의 처리 금지에 관한 사항
        verified: false        # 사람 검수 후 true
    triggers: {keywords: [...], patterns: []}
    absence_check: true
    note: ""                   # 선택, 1줄 이하. 서술형 코멘트 금지
```

- guidance 필드 폐지. norm_type·quote가 그 역할을 대체함.
- basis=practice(계약기간·완전합의 등 법령 근거가 없는 실무 항목)는 sources 빈 리스트 허용, UI에서 "실무" 배지로 법령 요건과 시각 구분.

## 빌드 검증 강화 (증적 강제 메커니즘)

- enrich: quote가 DB 조문 원문에 실제 존재하는지 부분문자열 대조(공백 정규화 후). 결과 3단계:
  - `quote_ok` — quote가 원문에서 확인됨 (+verified true/false에 따라 원문확인/원문 미대조 배지)
  - `quote_mismatch` — 조문은 찾았으나 quote 문언 불일치 → 빌드 경고 + 배지 (조작·오기 방지)
  - `missing` — 조문 자체 미발견 → 빌드 경고 + 배지
- validate: basis=statute인데 quote 없으면 빌드 실패.

## UI v2

- **주 탭 = 체크리스트 표**: 행 = 원자 항목. 열 = 매칭배지(✓제N조 추정 / ✗미검출 / —) · ID · check 질문 · 규범 · 근거(법령 조·항호 + 증적배지). 필터: 모듈/심각도/매칭상태. 행 클릭 → quote(강조) + 전체 조문 원문 + 매칭 조항 발췌.
- 분석 플로우(입력→유형→모듈 스크리닝)는 유지하되 결과가 이 표로 나옴. 미검출(absence_check ✗) 섹션을 표 상단에 고정.
- 조항별 뷰는 보조 탭으로 강등 (기존 split 뷰 재사용).
- 매칭 배지는 항상 "추정" 표기 — 판단은 전문가 몫임을 UI가 주장하지 않음.

## 콘텐츠

- 기존 28개 체크포인트를 항·호 단위로 원자 분해 → 약 80~120개. Phase 1 조번호 검증 성과(금소법 제45조 교체 등) 승계.
- 대량 생성 도구 `build/extract_candidates.py`: 열거형 조문("다음 각 호…포함하여야 한다")에서 호 단위 후보 YAML 자동 추출 (Phase 2 확장용 파이프라인, 산출물은 Claude 정제+사람 검수 필수). 실측: 관련 7개 법령에 열거형 조문 485건.
