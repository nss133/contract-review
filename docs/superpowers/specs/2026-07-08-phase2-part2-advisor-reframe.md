# Phase 2 Part 2: 검토 가이드 재정의 (채점기 → 검토 보조)

- 작성일: 2026-07-08 · 상태: 승인됨 (방향 확정, 구현 전)
- 배경: 사용자 피드백 — 앱이 무의식적으로 "합격/불합격 채점기"처럼 말한다("미검출", "누락 의심", 붉은 표식 대량 → "다 틀렸다"는 거부감). 실제 목적은 **"이 조항 볼 때 이런 걸 확인하세요"라고 짚어주는 전문가용 검토 보조**. 프레임 전환이 나머지 모든 항목의 답.

## 확정된 결정 (사용자 승인)
1. 상태 표시를 **짚음 / 확인 권장 / 검토 제안** 3분류로 (판정 언어 "미검출/누락" 폐기)
2. 심각도(필수/권장/참고)를 **근거 조문의 규범 효력에서 자동 도출** + 근거 명시
3. 스펙·계획 먼저

## 핵심 사실 (실측)
- 심각도 근거 미문서화: 현재 필수 48 / 권장 32 / 참고 8, 배정 기준이 어디에도 없음
- "단일후보 → 검토필요"는 도메인 착오: comp-match(법령 후보 택1)에선 의심 신호였으나, 계약서에선 "체크가 조항 하나에 매칭"이 정상. 조항을 펼치면 체크마다 "단일 후보 — 검토필요"가 줄줄이 떠 오독 유발
- 가이드라인·자율규제 근거 0건 (88개 전부 법령·감독규정만)

## A. 상태 모델 재정의 (tier → coverage)

기존 confirmed/review/none/missing → 검토 관점의 3+1 상태:

| 신규 상태 | 의미 | 기존 tier 대응 | 톤 |
|---|---|---|---|
| **짚음** | 계약서가 이 항목을 다루고 있음 ("제N조에 반영됨") | confirmed | 긍정(녹) |
| **확인 권장** | 관련 조항이 있으나 충분한지 눈으로 확인 권함 | review | 중립(청/황) |
| **검토 제안** | 이 항목이 안 보임 — "빠졌다"가 아니라 "필요한 계약인지 검토" | missing(재정의) | 주의(단, 조건부) |
| (조용한 기타) | 안 보이지만 저위험/저신뢰 → 알람 안 함 | none 비-alarm | 접힘/생략 |

## B. 알람(검토 제안)의 신뢰도·중요도 게이트

"미검출 메시지는 신뢰도를 높게 잡아야" 반영. **검토 제안은 아래를 모두 만족할 때만** 표면화:
- severity가 필수(또는 권장) — 저위험(참고) 항목의 부재는 알람 안 함
- 어떤 조항도 "확인 권장" 범위조차 못 든 확실한 부재
- (선택) 근거 조문이 강행규정
그 외 미매칭 필수 항목이라도 애매하면 "확인 권장"으로, 저위험은 조용한 기타로. → 계약기간 특정 같은 저위험 항목이 알람에서 빠짐.

## C. 단일후보 강등 제거 + reason 언어 재작성

- decideTier에서 **"단일 후보 → 검토필요" 강등 제거**. 계약서 도메인에선 단일 매칭이 정상. 짚음은 후보 2개 요구 없이 도달 가능하게 재튜닝(점수·인용·문구일치 기반).
- (골드셋·확정매칭 피드백 루프로 신뢰도를 시간에 따라 올리는 건 **후속 과제**로 명시. 지금은 "최대한 매핑"을 우선, 무섭게 표시하지 않기.)
- reason 문구를 판정형 → 정보형으로: "제5조 본문과 문구 일치", "핵심어 겹침", "명시 인용 일치", "관련 조항 있음 — 문구 확인 권장". "단일 후보/신호 — 검토필요" 삭제.

## D. 심각도 규범 효력 기반 자동 도출

- 각 check의 severity를 norm_type에서 도출: **강행(의무·금지)→필수 / 임의(권한)→권장 / 추정·간주·선언·정의·절차·실무→참고**.
- `severity_basis` 필드 추가(빌드 도출 또는 지식 명시): 예 "근거 조문이 강행규정(의무)". UI가 "이게 왜 필수?"에 이 문장으로 답. 검토 제안 알람에도 함께 제시.
- 기존 수동 배정(48/32/8)을 이 규칙으로 재계산 → 저위험 항목 자연 하향. 규칙과 어긋나 사람이 유지해야 할 예외는 명시적 override 필드로.

## E. 가이드라인·자율규제 근거 배선 (⑥)

- klia 협회규정(손해사정 업무위탁 모범규준, 보험권 표준내부통제기준 등, 조문 표제 보유)을 관련 check의 추가 source로. 기존 quote 검증 파이프라인 그대로.
- fsc_guidelines는 조 단위 아님 → 참고 링크로만(quote 앵커링 제외).
- 증적 배지에 자율규제 톤 추가 가능(법령 vs 자율규제 구분).

## F. UI 재설계

### 체크리스트 표
- 매칭 열 라벨: 짚음(✓ 반영) / 확인 권장(◑) / 검토 제안(!) / 기타(접힘). 붉은 "✗ 미검출" 폐기.
- 정렬: 검토 제안(중요·신뢰 높은 것) → 확인 권장 → 짚음 → 기타. 단 "짚음"도 접근 쉽게.
- 각 행에 severity + 근거 한 줄.

### 조항별 좌우 대비 (⑧)
- 조항을 펼치면 2열: **좌 "이 조항에서 반영된 검토항목"(짚음)** / **우 "추가 확인 제안"(확인권장·관련)**. 잘 된 것과 볼 것을 나란히. 평소 칸은 축소, 펼쳐보기.

## G. 리포트 재설계

현재 "누락 의심 대량 나열 + 정체불명 체크박스" → 검토 워크시트로:
- **긍정 먼저**: 상단 요약 타일 "짚어진 항목 N / 확인 권장 M / 검토 제안 K". 실패 목록이 아니라 검토 진행 현황.
- 검토 제안 각 항목에 **심각도 근거**를 붙여 "왜 봐야 하는지" 제시(D 활용).
- 체크박스 → **검토자 확인(sign-off)**으로 의미 명확화("확인함") 또는 제거.
- 조항별 좌우 대비(F)의 요약을 리포트에도 반영해 "이 계약서에서 뭘 잘 다뤘고 뭘 더 볼지"가 한눈에.

## 비범위 (후속)
- 골드셋·확정매칭 피드백 루프(신뢰도 학습) — 방향만 명시, 이번 미구현
- 유형 확대(주주간계약·합의서 등, ④), 표준계약서 수집(⑤)
- 임베딩(폐기 유지)

## 검증 기준
- 저위험 항목(계약기간 특정)이 검토 제안 알람에서 빠지는지
- 조항 펼침에서 "단일 후보 — 검토필요" 문구가 사라지고 정보형 reason으로 바뀌는지
- 심각도 각 항목에 근거 문장이 붙는지, 필수가 강행규정 근거를 가지는지
- 가이드라인 근거가 실제 표시되는지
- 리포트가 긍정-먼저 워크시트로 읽히는지 (브라우저 육안)
